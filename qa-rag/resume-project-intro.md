# 企业级 AI Agent 服务 — 简历项目介绍

## 基本信息

- **项目名称**：Enterprise AI Agent — 企业级 AI Agent 服务
- **技术栈**：FastAPI / Python 3.11 / LangChain / LangGraph / OpenAI / Milvus / Redis / PostgreSQL / SQLAlchemy / sentence-transformers / Langfuse / Docker
- **项目定位**：面向生产环境的多能力 AI Agent 服务骨架，提供 Agent 编排、RAG、记忆、意图识别、工具调用等核心能力，及配套的生产级基础设施与质量保障体系。

---

## 核心亮点

### 1. 双模式 Agent 编排引擎

支持 **ReAct**（推理-行动-观察循环）和 **Plan-and-Execute**（先规划再执行）两种 Agent 模式，通过意图识别结果动态选择策略。Plan 模式执行失败时自动降级回退到 ReAct，保证服务可用性。

意图识别基于关键词匹配 + 置信度评分：命中公式 `min(0.95, 0.45 + 0.12 * hits)`。低于阈值（默认 0.55）时触发澄清流程，提示用户明确需求。

### 2. 多路 RAG 检索管道

实现三模检索：
- **向量检索**：Milvus ANN 搜索（L2 距离）
- **关键词检索**：BM25 Okapi（k1=1.5, b=0.75）
- **混合检索**：两者并行 + RRF 融合（Reciprocal Rank Fusion, k=60）

融合结果经 **Cross-Encoder** 深度模型（`cross-encoder/ms-marco-MiniLM-L-6-v2`）二次重排序，显著提升召回精度。配套完整的文档 ETL 流水线，支持 TXT/PDF 解析和三种分块策略（Fixed / Recursive / Paragraph）。

### 3. 双层记忆系统

| 维度 | 短期记忆 | 长期记忆 |
|------|---------|---------|
| 存储 | Redis List | Milvus Collection |
| 容量 | 20 条 / 4000 tokens | 无上限 |
| 超量策略 | LLM 自动摘要压缩 | — |
| 检索 | 顺序读取 | 语义相似度召回 |
| 用途 | 会话连贯性 | 跨会话知识复用 |

短期记忆超出消息数或 Token 阈值时，自动调用 LLM 生成摘要替换历史消息。长期记忆支持按 session 维度的语义召回和主动遗忘。

### 4. 生产级基础设施

- **多模型路由**：优先级调度 + 加权随机负载均衡 + 自动降级
- **熔断器**：三状态（CLOSED / OPEN / HALF_OPEN），5 次失败阈值，60s 恢复超时，防止 LLM 故障扩散
- **语义缓存**：基于余弦相似度（阈值 0.95）实现近似查询命中，无 Embedder 时降级为 bigram hash + Jaccard 相似度
- **链路追踪**：接口抽象（Tracer），支持对接 Langfuse 等可观测平台

### 5. 可观测性与离线评测

- 接入 **Langfuse**，实现 LLM 调用链路追踪、Token 用量监控、Bad Case 排查
- 搭建 **离线评测框架**，基于测试数据集自动化评估 RAG 忠实度、答案相关性、上下文精度，防止质量回归

### 6. 输出质量反思

Agent 回答后由独立的 **ReflectionAgent** 进行质量审查，从完整性、幻觉风险、证据一致性等维度由 LLM 评分（0-100），低于 60 分触发告警，为自动重试或人工介入提供依据。

## 架构设计

```
┌─────────────────────────────────────────────────┐
│                  接入层                          │
│     FastAPI Routes (chat / document / health)    │
├─────────────────────────────────────────────────┤
│                  领域核心层                       │
│   Agent 编排  │  RAG 管道  │  记忆  │  工具/意图   │
│   (ReAct/Plan)│ (检索/重排) │ (短/长) │ (注册/路由)  │
├─────────────────────────────────────────────────┤
│                 基础设施层                        │
│  LLM 路由/熔断 │ 向量库  │ 缓存  │ DB  │ 可观测性   │
├─────────────────────────────────────────────────┤
│              数据 / ETL 层                       │
│         文档解析 → 分块 → 向量化 → 入库           │
└─────────────────────────────────────────────────┘
```

**设计原则**：分层抽象（每层通过 Protocol 接口依赖）、配置驱动（pydantic-settings）、降级优先（多级降级保证核心可用性）。
